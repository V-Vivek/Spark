
# Spark Driver Properties Configuration

## Overview
Spark driver properties control the resources and behavior of the driver program, which orchestrates the Spark application execution.

## Key Driver Properties

### Memory Configuration
```properties
spark.driver.memory=4g
```
- Controls heap size for the driver JVM 
- Critical for collecting results(collect and take actions) and managing metadata(DAG, task scheduling)

### Core Allocation
```properties
spark.driver.cores=2
```
- Number of CPU cores allocated to the driver
- Affects parallel task scheduling and coordination



## Significance
1. **Application Stability**: Prevents driver OOM errors when collecting large datasets
2. **Performance**: Adequate cores enable efficient task scheduling and DAG computation
3. **Resource Planning**: Proper sizing prevents cluster resource waste
4. **Scalability**: Ensures driver can handle metadata for large-scale applications
5. **Fault Tolerance**: Correct network settings enable reliable executor-driver communication

## Best Practices
- Set `spark.driver.memory` based on data collection needs (typically 2-8GB)
- Monitor driver metrics to tune configurations
- Use `spark.driver.maxResultSize` to limit collected data size
- Consider cluster mode vs client mode deployment requirements



# Spark Executor Properties

### Memory Configuration
```properties
spark.executor.memory=4g
```
- Heap memory allocated to each executor JVM
- Used for caching data, shuffle operations, and task execution
- Does not include memory overhead

### Core Allocation
```properties
spark.executor.cores=4
```
- Number of CPU cores per executor
- Determines task parallelism within each executor
- More cores = more concurrent tasks per executor

### Executor Count
```properties
spark.executor.instances=10
# or use dynamic allocation
spark.dynamicAllocation.enabled=true
```
- Total number of executors to launch
- Controls overall application parallelism
- Trade-off between resource utilization and overhead

### Memory Overhead
```properties
spark.executor.memoryOverhead=512m
```
- Off-heap memory for JVM overhead, native libraries, and internal metadata
- Defaults to 10% of executor memory (minimum 384MB) whichever is greater
- Critical for containerized environments (YARN, Kubernetes)

## Scenario
Cluster Configuration:
10 Nodes 
16 cores per Node
64GB memory per Node

### Configuration Example: Bad vs Good

#### ❌ Bad Configuration
```properties
spark.executor.instances=150
spark.executor.cores=1
spark.executor.memory=4g
```

**Problems:**
- **Cores per executor**: 1 core leads to poor HDFS throughput and inefficient CPU usage
- **Executors per node**: 16 executors per node (16 cores - 1 for OS) = 15 executors per node
- **Memory per executor**: (64GB - 1GB for OS) ÷ 15 ≈ 4.2GB → 4GB heap + 0.4GB overhead = 4.4GB per executor
- **Total executors**: 10 nodes × 15 = 150 executors
- **No buffer retained**: Leaves no room for system processes or YARN/K8s

**Actual Utilization:**
- Total cores: 150 × 1 = 150 cores (93% utilization)
- Total memory: 150 × 4.4GB = 660GB (109% utilization - leads to OOM)
- Poor parallelism: 150 concurrent tasks but with high overhead and contention

**Performance Issues:**
- Large heap sizes lead to long GC pauses
- Inefficient HDFS I/O due to single-core executors
- High risk of executor failures due to memory pressure


#### ✅ Good Configuration
```properties
spark.executor.instances=30
spark.executor.cores=5
spark.executor.memory=21.3g
spark.executor.memoryOverhead=2.1g
```

**Rationale:**
- **Cores per executor**: 5 cores improves HDFS throughput and CPU efficiency
- **Executors per node**: 16 cores - 1 for OS = 15 cores available; 3 executors per node (5 cores each)
- **Memory per executor**: (64GB - 1GB for OS) ÷ 3 ≈ 21GB → 21.3GB heap + 2.1GB overhead = 23.4GB per executor
- **Total executors**: 10 nodes × 3 = 30 executors

**Actual Utilization:**
- Total cores: 30 × 5 = 150 cores (93% utilization)
- Total memory: 30 × 23.4GB = 702GB (91% utilization)
- Better parallelism: 5 concurrent tasks per executor, compared to 1 in bad config

**Performance Benefits:**
- Smaller heap sizes result in faster, more frequent GC cycles
- Better task-level parallelism across more executors
- Improved fault tolerance
- Optimal HDFS I/O throughput per executor


# Other Important configurations

## `spark.default.parallelism`
```properties
spark.conf.set("spark.default.parallelism", "100")
```
- Only applies to RDDs
- Sets default number of partitions for RDDs
- Default is total number of executor cores on all nodes of the cluster
- Typically set to 2-3 times the total number of executor cores
- Wide Transformations (like join, reduceByKey) create partitions based on this setting during shuffle operations



## `spark.sql.shuffle.partitions`
```properties
spark.conf.set("spark.sql.shuffle.partitions", "100")
```
- Only applies to Spark SQL and DataFrame operations
- Sets number of partitions for shuffle operations
- Default is 200; adjust based on data size and cluster resources
- Higher values can improve parallelism but increase overhead
- Wide Transformations (like join, reduceByKey) create partitions based on this setting during shuffle operations
