# Caching in RDD

## What is Caching in RDD?

- To store the intermediate results of an RDD in memory, so that future actions on this RDD can be executed faster.
- By default, RDDs are recomputed each time an action is performed. 
- Caching avoids this recomputation by keeping the data in memory.
- You can mark an RDD to be persisted using the persist() or cache() methods on it


## Example Use Case

Suppose you are analyzing a large log file to extract error messages and then perform multiple actions like counting, filtering, and aggregating.

```python
# Read log file into RDD
logs = spark.sparkContext.textFile("hdfs://path/to/logfile")

# Filter error messages
errors = logs.filter(lambda line: "ERROR" in line)

# Cache the filtered RDD
# Here RDD cache is not created yet, this is just a lazy transformation
errors.cache()

# Perform multiple actions
error_count = errors.collect() # Actual caching happens here
error_lines = errors.count() # Uses cached data
```

**Explanation:**
- The `errors` RDD is marked for caching after filtering.
- Actual caching occurs when the first action (`collect()`) is called.
- Subsequent actions (`count()`, etc.) use the cached data, speeding up execution.


## Behavior of Caching with Different Actions
```python
from pyspark.sql import SparkSession
spark = SparkSession.builder.master("local[*]").getOrCreate()

# Create RDD with 4 partitions
rdd = spark.sparkContext.parallelize(range(1000), 4)
filtered_rdd = rdd.filter(lambda x: x % 2 == 0)

# Cache the RDD (lazy operation)
filtered_rdd.cache()

print("=== First Action: take(5) ===")
result1 = filtered_rdd.take(5)
print(f"Result: {result1}")
# Only Partition 0 (and maybe Partition 1) are cached

print("\n=== Second Action: collect() ===")  
result2 = filtered_rdd.collect()
print(f"Total records: {len(result2)}")
# Uses cached partitions + computes remaining partitions
```
## **When `take(5)` is Called (First Action)**

When `take(5)` is executed on a cached RDD:

- **Only the data needed to satisfy `take(5)` is computed and cached**
- Spark processes partitions **one by one** until it finds 5 records
- **Only the partitions that were read are cached**, not the entire RDD
- The caching is **partial** and **partition-level**

## **When `collect()` is Called (Second Action)**

When `collect()` is executed:

- Spark uses **whatever data is already cached** from the previous `take(5)` operation
- For partitions **not yet cached**, Spark will:
  1. Compute the missing partitions
  2. Cache them as they are computed
  3. Return all data (cached + newly computed)




## **Detailed Flow**

### **Partition Layout (Example)**
```
Partition 0: [0, 1, 2, ..., 249]     -> After filter: [0, 2, 4, ..., 248]
Partition 1: [250, 251, ..., 499]   -> After filter: [250, 252, ..., 498] 
Partition 2: [500, 501, ..., 749]   -> After filter: [500, 502, ..., 748]
Partition 3: [750, 751, ..., 999]   -> After filter: [750, 752, ..., 998]
```

### **After `take(5)`:**
```
✅ Partition 0: CACHED (contains first 5 even numbers: 0,2,4,6,8)
❌ Partition 1: NOT CACHED (not needed for take(5))
❌ Partition 2: NOT CACHED (not needed for take(5))  
❌ Partition 3: NOT CACHED (not needed for take(5))
```

### **After `collect()`:**
```
✅ Partition 0: USES CACHE (already cached)
✅ Partition 1: COMPUTES + CACHES (newly computed and cached)
✅ Partition 2: COMPUTES + CACHES (newly computed and cached)
✅ Partition 3: COMPUTES + CACHES (newly computed and cached)
```

## **Key Points**

1. **Caching is partition-level**, not record-level
2. **`take(5)` only caches the partitions it needs** to find 5 records
3. **`collect()` uses cached partitions** and computes missing ones
4. **Subsequent actions** will use the fully cached RDD

## **Storage Levels**
- cache() is equivalent to persist(StorageLevel.MEMORY_ONLY)
- You can choose different storage levels based on your use case:

| Storage Level                | Description                                                                 | Use Case / Notes                                 |
|------------------------------|-----------------------------------------------------------------------------|--------------------------------------------------|
| MEMORY_ONLY                  | Stores RDD as deserialized Java objects in memory.                          | Fastest, but data is recomputed if not enough RAM|
| MEMORY_AND_DISK              | Stores RDD in memory, spills the remaining partitions to disk if not enough RAM.                     | Good for large RDDs that don't fit in memory     |
| DISK_ONLY                    | Stores RDD only on disk.                                                    | Use when memory is very limited                  |
| OFF_HEAP (experimental)      | Stores RDD in off-heap memory (outside JVM heap).                           | Requires extra configuration                     |


```python
# Different storage levels affect this behavior
from pyspark import StorageLevel

# Memory only (default for cache())
rdd.persist(StorageLevel.MEMORY_ONLY)

# Memory and disk (spillover)
rdd.persist(StorageLevel.MEMORY_AND_DISK)
```

## Unpersisting an RDD

- Use `unpersist()` to remove an RDD from memory and disk when caching is no longer needed.
- Although Spark automatically manages memory, explicitly unpersisting can free up resources sooner.
- By default, `unpersist()` is asynchronous. Meaning it returns immediately and the actual unpersisting happens in the background.
- To block or wait until unpersisting is complete, use `rdd.unpersist(blocking=True)`.

### Example

```python
# Cache an RDD
rdd.cache()
rdd.count()  # Triggers caching

# ... perform actions ...

# Remove from cache when done
rdd.unpersist()

# Or to block until unpersisting is complete
rdd.unpersist(blocking=True)

# ... perform other operations ...
```
